# Contributing to UTP

Thanks for your interest in the Universal Transfer Protocol! We welcome contributions from everyone.

---

## How to Contribute

### 1. **Report Issues or Suggest Improvements**

Found a bug or have an idea? [Open an issue](https://github.com/utp-foundation/utp-spec/issues).

**Include:**
- What you found (or what you think should be added)
- Why it matters
- If applicable, specific examples or use cases

### 2. **Join the Discussion**

Have questions or want to discuss UTP? [Start a discussion](https://github.com/utp-foundation/utp-foundation/discussions).

- Ask questions
- Share your implementation
- Discuss use cases
- Propose new domains

### 3. **Submit Pull Requests**

Want to contribute code or content?

**For small changes (typos, clarifications):**
1. Fork the repo
2. Edit the file
3. Submit a PR with a clear description
4. We'll review and merge

**For large changes (new domain profiles, major spec updates):**
1. Open an issue first to discuss
2. We'll agree on approach
3. Then submit a PR

### 4. **Propose a New Domain**

Want to define UTP for a new industry (healthcare, legal, travel, etc.)?

1. [Open an issue](https://github.com/utp-foundation/utp-spec/issues) titled: `[Domain Proposal] Industry Name`
2. Include:
   - Industry/domain name
   - Why agents need authorization in this domain
   - Example use cases
   - Initial thoughts on action codes or signals
3. We'll discuss and help you build it
4. Once complete, add to `/domains/` folder

### 5. **Build an Implementation**

Built UTP in your language or platform?

1. Make sure it follows the spec (v1.0)
2. [Open an issue](https://github.com/utp-foundation/utp-spec/issues) or [discussion](https://github.com/utp-foundation/utp-spec/discussions) to tell us about it
3. We'll add it to the "Known Implementations" list in the README

## Types of Contributions

### Specification Updates
- Clarifications to confusing sections
- Examples or use cases
- Corrections to errors
- Architectural improvements (with discussion first)

### Domain Profiles
- New industry profiles (healthcare, legal, travel, etc.)
- Action code dictionaries
- Scope boundaries
- Enforcement rules

### Implementations
- Code in any language (Node.js, Python, Go, Rust, Java, etc.)
- Reference implementations
- Integrations with banks, fintechs, or data aggregators

### Documentation
- Blog posts about UTP
- Tutorials or guides
- Comparison analyses
- Regulatory alignment docs

### Feedback
- Bug reports
- Security issues (see below)
- User experience feedback
- Questions from your implementation

## Code of Conduct

Be respectful, inclusive, and constructive. We value:
- Diverse perspectives
- Good-faith debate
- Collaborative problem-solving
- Respectful disagreement

---

## Security Issues

**Do not open a public issue for security vulnerabilities.**

Email [security@utp.io](mailto:security@utp.io) with:
- Description of the vulnerability
- Steps to reproduce (if applicable)
- Potential impact

We'll investigate and work with you on a fix before public disclosure.

---

## PR Guidelines

### Before You Submit

- Read the [UTP Specification](./SPEC.md)
- Make sure your change aligns with the spec and philosophy
- Test your changes (if applicable)

### When You Submit

**Title:** Clear, descriptive (e.g., "Add healthcare domain profile" or "Clarify Stage 7 resolution logic")

**Description:**
- What is this change?
- Why is it needed?
- Any alternatives considered?
- For spec changes: impact on implementations

**Link issues:** If your PR addresses an issue, reference it: "Closes #123"

### Review Process

1. We'll review and may ask questions or suggest changes
2. We'll discuss implementation approach
3. Once approved, we merge
